# RAG Project — Decisions Log

Project: Clinical RAG assistant for nephrologists/fellows, built on the KDIGO 2025 IgAN/IgAV Clinical Practice Guideline (executive summary PDF).

Status: portfolio/learning project, scoped to one document now, architected to extend to the full KDIGO guideline library later. Designed so multi-document support requires no rework — `source`/`doc_type` metadata exist from day one even with a single document indexed.

This file is the single source of truth for every locked-in decision. Update it after each phase is finalized. Any AI tool (Claude, Cursor agent, etc.) working on implementation should read this file first.

## Phase 0 — Project Design (COMPLETE)

### 0.1 Success Metrics

| Target | Value | Rationale |
|---|---|---|
| Retrieval (primary) | Recall@10 ≥ 95% | What matters is the correct chunk reaching the LLM's context, not its rank position — the LLM reads all top-k chunks, not just #1. |
| Retrieval (diagnostic, not gating) | MRR — tracked, no fixed target | Signals whether reranking is working and whether "lost in the middle" risk exists if correct chunks land late in the list. |
| Answer quality | LLM-judge score ≥ 4.5 (accuracy / completeness / relevance, 1–5 scale) | Clinical decision support — wrong or incomplete answers have real consequences. |
| Latency | ≤ 5s total, response streamed, first token < 1s | Chosen deliberately over a stricter ≤2s budget specifically to leave room for LLM-based reranking (added latency ~500–1000ms), which was prioritized as a learning goal over raw speed. |
| Cost | ≤ $20/month | Guardrail for a single-document portfolio project; not a hard architectural constraint but a sanity check on model/service choices. |

### 0.2 Document Inventory

| Attribute | Value |
|---|---|
| Format | Digital PDF, embedded/selectable text — no OCR required |
| Structure | Well-structured: clear section headers (Diagnosis, Treatment, Special situations, etc.), 1 dense multi-column table (Table 1), 2 flowchart figures with embedded text labels (Figure 1, Figure 2) |
| Size | 7 pages total; 6 usable (references page excluded) |
| Known extraction risks | Table 1 — multi-column table, risk of scrambled row/column order on naive text extraction. Figures 1 & 2 — flowchart text may extract without preserving visual/logical relationships between boxes; flagged as a known gap for Phase 1.1, options to resolve later: manual transcription, skip, or vision-LLM description. |
| Noise to exclude from indexing | References list, author disclosures/conflict-of-interest block — not clinically actionable content, should not be chunked or embedded |
| Update frequency (real-world) | Every couple of years |
| Maintenance strategy | Manual trigger (you add/update a document by hand) + incremental update (only the new/changed document is re-processed; existing chunks for unchanged documents are left untouched). Full rebuild and automated event-driven re-indexing were both considered and rejected as overkill for this update frequency and trigger model. |

### 0.3 Question Taxonomy + 0.5 Test Set

Six categories used (generic retrieval-stress patterns, not medical-specific — applicable to legal, financial, or any factual/reference-style RAG project):

1. Direct fact
2. Numerical / threshold
3. Spanning
4. Comparative
5. Temporal
6. Holistic

Test set: 30 questions (5 per category), JSONL format, schema:
```json
{"id": "...", "category": "...", "question": "...", "reference_answer": "...", "keywords": ["...", "..."], "source_section": "..."}
```

File: `test_set.jsonl` — generated by Claude based on actual KDIGO guideline content, **not yet manually verified line-by-line against the source document** (flagged as a known gap — accepted for this learning project; would require verification in a real production system since the test set is the ground truth all later metrics are measured against).

Note on `keywords` field: NOT used for retrieval. Used only in evaluation (Phase 3.2) — after top-k chunks are retrieved, check what fraction of expected keywords appear in the retrieved text. This is a cheap, fuzzy complement to Recall@k (which requires a pre-identified correct chunk per question).

## Phase 1 — Ingestion Pipeline (IN PROGRESS)

### 1.1 Document Extraction and Cleaning (COMPLETE)

Design lens for all of Phase 1: even though this project currently indexes a single PDF, every ingestion decision is made as if a folder of many guideline PDFs will be dropped in later. No document-specific hacks (e.g., hardcoded marker strings, one-off table handling) — every step must be a general rule that happens to apply to this document, not a rule written for this document.

| Sub-decision | Choice | Rationale |
|---|---|---|
| Extraction library | **pdfplumber** | Gives both text extraction and structural table detection in one library — one consistent extraction path across all future documents rather than per-document tooling choices. |
| Table handling | **General table-detection step**: run `find_tables()` on every page of every document. Any detected table is pulled out and processed separately from surrounding body text, then linearized into self-contained text (column headers repeated per row so each row stands alone) before being rejoined into the document's text flow at the point it occurred. | Not a "Table 1" special case — this fires on whatever tables exist in whatever document is being ingested. Prevents scrambled row/column extraction (the known risk flagged in 0.2) generally, not just for this table. |
| Figure/image handling | **General image-detection step**: detect embedded images/figures on each page (via pdfplumber page images or bounding-box crops). Each figure image is sent to a cheap vision-capable LLM (GPT-4o-mini or Claude Haiku tier) with a prompt to describe its content in plain prose — including all text labels and the logical/flow relationships between elements. The description is inserted inline into the extracted text at the point the figure occurred, clearly tagged (e.g., `[Figure 1 description: ...]`). | Manual transcription doesn't scale to many documents; skipping loses real clinically-actionable content (confirmed needed for test-set questions q14, q16). Vision-LLM description is the standard scalable approach and costs a few cents per document at ingestion time (one-time cost, not per-query). |
| Noise exclusion | **Config-driven section-header exclusion list**, checked against detected section headers during extraction. Any section whose header matches an entry in the list (e.g., `["DISCLOSURE", "REFERENCES", "CONFLICT OF INTEREST", "ACKNOWLEDGMENTS"]`) is dropped from the cleaned output. List lives in a config file, not inline code. | Hardcoding a truncation marker specific to this PDF breaks on the next document with different section names. A config-driven exclusion list means adding a new document type is a config edit, not a code change. |

**Implementation note for Cursor agent:** build this as a pipeline of discrete steps (extract text → detect/linearize tables → detect/describe figures → apply section exclusion list → output cleaned text) so each step is independently testable and reusable across documents. Output of 1.1 lands in `data/cleaned/`, one cleaned text file per source PDF, ready for Phase 1.2 chunking.
